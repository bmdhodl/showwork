# Architecture

showwork is outcome verification for AI agents. An agent session records falsifiable claims about what it changed; showwork verifies those claims against reality and refuses to bless a "done" that is not real. Observability logs what an agent did. showwork proves what it claimed.

This document describes how the pieces fit. `SPEC.md` is the normative ledger contract; when the two disagree, the spec wins.

## Two layers, one dialect

The system is deliberately split into a writer side and a reader side:

1. **Python reference implementation** (`src/showwork/`, published to PyPI as `showwork`). The full read-write engine: records sessions, claims, retractions, and finish events; re-executes checks; computes verdicts; maintains and audits the integrity chain. Stdlib only, zero runtime dependencies.
2. **JS reader-side auditor** (`js/showwork-audit/`). A zero-dependency Node implementation of the reading half of the spec. It audits integrity chains and computes verdicts, but re-executes no checks. Per the spec's reader-only conformance clause, everything it does not re-verify is reported, never silently skipped.

Both layers speak the same frozen ledger dialect: `spec-v0.2` JSONL framing, line segmentation, and chain semantics. The shared chain fixtures under `tests/fixtures/chain/` (generated by `scripts/make_chain_fixtures.py`, expected verdicts in `expected.json`) hold both implementations to identical verdicts. `js/showwork-audit/test.mjs` and the Python suite both run against them. Changing the dialect means changing `SPEC.md`, the fixtures, and both implementations in the same commit; treat that as last resort until v1.

## Data flow

```
session start
   -> claims (falsifiable checks appended to .showwork/)
      -> verification (each check re-executed against the filesystem / locked commands)
         -> audit (integrity chain over every appended record)
            -> verdict (GREEN | YELLOW | RED)
               -> exit gate (finish --status ok refuses on RED, exit 2)
```

Concretely: session -> claims -> audit -> verdict.

- **Ledger** (`.showwork/` in the target repo): `sessions.jsonl` for lifecycle events, `claims-YYYY-MM-DD.jsonl` for claims and retractions. Append-only. Corrections are retraction records that reference the original claim; history is never rewritten.
- **Claims** carry one of six check types: `file_exists`, `file_contains`, `path_moved`, `frontmatter`, `glob_count`, `command`. Free-form prose without a check is recorded but never counts as verified.
- **Verification is deterministic.** Checks re-run against the filesystem or a locked command. No LLM judges an LLM.
- **Integrity chain**: every appended record carries a `prev` hash extending a per-file hash chain from a genesis anchor. Audits detect tampering, deletion, and unchained appends, and accept concurrent forks (a `prev` re-anchored to an earlier line) as GREEN while reporting them.
- **Verdict algebra**: RED when an active RED-severity claim fails; YELLOW when a YELLOW claim fails or a checker errors; GREEN otherwise.

## Module map (Python reference)

| Module | Role |
|---|---|
| `src/showwork/ledger.py` | Append-only ledger: start/claim/finish/retract, chain writing, session and date verification |
| `src/showwork/checks.py` | The six checkers, anti-vacuous rules, retraction application, verdict evaluation, report rendering |
| `src/showwork/audit.py` | Integrity-chain audit: per-file and per-root chain verification |
| `src/showwork/cli.py` | `showwork` CLI: `start`, `claim`, `retract`, `verify`, `finish`, `audit`, `run`, `stop-hook` |
| `src/showwork/hooks.py` | Stop-hook adapter: observes and records a verdict when an agent stops; never blocks |
| `js/showwork-audit/index.mjs` | Reader-side chain audit and verdict computation in Node |

Supporting scripts (operator tooling, not part of the package API): `scripts/run_tests.py` (behavioral + conformance suite), `scripts/evidence_pack.py` (auditor-readable evidence pack from a chain-verified ledger; refuses RED chains), `scripts/derive_case_study.py` and `scripts/false_done_rate.py` (sanitized metrics bound to source ledgers).

## Exit gate vs Stop hook

Two distinct surfaces, kept distinct on purpose (the spec requires it):

- **Exit gate** (`showwork finish --status ok`): verifies the session's own claims first. Any RED refuses the close with exit 2. A `--no-verify` bypass is stamped on the event and fails the CI gate. This is the enforcement point.
- **Stop hook** (`showwork stop-hook`, adapter in `docs/claude-code.md`): fires when an agent session stops, records the verdict as telemetry, never blocks the host process. Observation, not enforcement.

## CI surface

- `actions/verify/` is a composite GitHub Action that gates a job on receipts: a broken chain, a RED claims verdict, or a `--no-verify` close fails the job (`strict: true` also fails YELLOW). `allow-commands` defaults to false so fork PRs never execute repo code; command claims then degrade honestly to YELLOW.
- The repo gates itself with it (`.github/workflows/ci.yml` on a self-hosted runner): the behavioral suite, the JS conformance run against the frozen fixtures, and a receipts job that verifies the repo's own committed sessions. The committed `.showwork/` ledger verifying in CI is the live demo.

## Public surface

- PyPI package `showwork` (0.2.0, `requires-python >= 3.10`, zero runtime deps), MIT license.
- `SPEC.md`: the portable `spec-v0.2` ledger specification. Every normative requirement names a behavioral test beside it; conformance is defined there, including reader-only conformance for auditors like `js/showwork-audit`.
- `docs/`: adapters, CI gating, Claude Code Stop hook, fleet adoption, compliance mapping, case study with derived metrics. agent47 wires the same receipt pattern into its own repo sessions (agent47 PR #648).
- Receipts for this repo's own development ship in `.showwork/` with each PR; the repo eats its own dog food (`AGENTS.md`).

## Design invariants

1. Claims are falsifiable or they are just prose.
2. Verification is deterministic; no model in the loop.
3. The ledger is append-only; retraction, never rewrite.
4. Zero runtime dependencies in both implementations.
5. `SPEC.md` plus its named tests are the contract; ledger-format changes are breaking until v1.
